# KB Conventions

This file is the contract for how every document in the local knowledge base is structured. It exists so that a human editing in Obsidian and the indexing builder (`bin/relation-build`) agree on the same shape, and so that a new project cloned from this pattern starts with the rules already written down. Read it once before adding documents; the builder reads the machine-relevant parts on every run.

## Where documents live

All knowledge-base content lives under `kb/`. That directory is the Obsidian vault and the single source of truth. Inside `kb/`, the top level is one folder per collection, and documents nest in sub-folders that mirror how they were organised as parent/child pages. A document is a single Markdown file with a `.md` extension and a filesystem-safe, wikilink-friendly slug for a name.

Files outside `kb/` — logos, decks, PDFs, the `bin/` scripts, `.ai/` machinery — are **not** knowledge base and are not governed by this contract.

## Frontmatter: the per-document contract

Every document opens with a YAML frontmatter block. It has two zones with different owners, and the distinction matters: if you blur them, the builder will overwrite your edits or you will corrupt the index.

```yaml
---
# ── human-owned: you write and maintain these ──
title: "Example Document Title"
type: strategy                       # see the type list below
status: working                      # see the status list below
tags: [example, tag]                 # optional, your own keywords
relates-to: [other-document-slug]    # optional, slugs of related docs
ai-note: "How the AI should use this document."   # optional, a hint to the AI

# ── builder-owned: regenerated by bin/relation-build, never hand-edit ──
ai:
  domains: [strategy, product]
  topics: [example, topic]
  entities: { people: [], orgs: [], regulations: [] }
  summary: "A synthesised summary of what this document is about."
  generated: 2026-01-01
  source: local:kb/collection/example-document.md
---
```

### Human-owned fields

You own everything above the `ai:` key. The builder never touches these.

- **`title`** — the real, human title of the document. Quote it if it contains a colon or apostrophe.
- **`type`** — what kind of document this is. One of: `strategy`, `intelligence`, `bizdev`, `custdev`, `operations`, `finance`, `reference`, `decision`. This is the document's nature, which is usually but not always the same as its collection folder; when they differ, `type` wins as the description of the content.
- **`status`** — how settled the document is. One of: `draft` (sketch, not trustworthy yet), `working` (actively in formation, the live edge of thinking), `authoritative` (settled, safe to cite and build on), `archived` (kept for history, superseded).
- **`tags`** — optional free-form keywords in your own vocabulary. These are yours, separate from the builder's `topics`.
- **`relates-to`** — optional list of slugs of documents this one leans on or echoes, used to make relationships explicit beyond inline links.
- **`ai-note`** — optional one line addressed to the AI: how to use this document, what it anchors, what to watch for. This is the place to say "treat this as canonical for X" or "numbers here are estimates."

### Builder-owned fields (`ai:` block)

Everything under `ai:` is written and overwritten by `bin/relation-build`. **Do not hand-edit it.** It is the structured index the retrieval layer reads: `domains` and `topics` for ranking, `entities` (people / orgs / regulations) for entity lookups, a one-line `summary`, the `generated` date, and `source` provenance. If you change the body of a document, the `ai:` block goes stale until the builder reruns — that is expected; the builder refreshes it.

The one field worth understanding is **`source`**, the provenance of the document:
- `local:<path>` — authored locally; the named file is canonical (e.g. `local:kb/strategy/decision-log.md`).
- An optional **`supersedes:`** line records when a local-canonical document deliberately replaces a stale source, so the history is not silently lost.

## Links

Internal references between documents use Obsidian wikilinks keyed on the target's slug: `[[target-slug]]`. A link to a document that does not exist yet is allowed — it marks an intended connection, not an error. External references are ordinary Markdown links to real, clickable URLs.

## Writing for the reader, not the machine

Document bodies are written for a person. Keep internal machine identifiers — doc ids, decision codes, relation checksums — out of the human prose, because they mean nothing to a reader and break the flow. When you need to record them, collect them in a clearly-labelled **"For AI / source map"** section at the end of the document, and make anything a human might want to follow a real link rather than a bare code. The audience sets the vocabulary: know your reader and write in their language.

## The index

`kb/_index.md` is a single generated map of the whole base — one line per document (linked title, one-line summary, type and tags), grouped by collection, with a domain cross-reference. It is produced by the builder and replaces the need to open dozens of files to find your way around. Do not hand-edit it; rerun the builder to refresh it.

---

*For AI / source map: frontmatter schema and field semantics defined here are consumed by `bin/relation-build` (writes the `ai:` block and `kb/_index.md`) and by the `kb-retriever` agent (reads `ai:` topics/domains/entities for ranking).*
