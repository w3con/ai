## Task statement

Сделать операционную парадигму живой и дать ей дом на диске. Сейчас парадигма существует только
как черновик-спека (`CLAUDE.draft.md`) и обсуждения (`decisions.md`), а боевой `~/.claude/CLAUDE.md`
их не несёт; сессий и KB на диске нет вообще. Этот план: (1) записывает крисп-парадигму в боевой
`~/.claude/CLAUDE.md` на английском; (2) заводит на диске структуру `kb/` + `session/` + `decisions/`
с шаблонами, спроектированную так, чтобы будущий сырой вектор-стор встал рядом без переделки;
(3) переякоривает гранулярные memory-факты под парадигму и переводит их на английский. После этого
«начать пользоваться» становится возможным: появляются сессии, KB и самодостаточные правила.

**Дизайн-принцип «вожжи и лошадь» (Alex, 2026-06-21).** План даёт Alex явные **вожжи** — точки
контроля на **скобках** (команды `/_n`, гейт «Выполняй план», одобрение плана). Но **середину**
loop'а механизировать НЕ надо: CLAUDE.md (ум лошади) должен позволять мне ехать самому, следуя
LOOP'у без того, чтобы Alex правил каждый шаг. Поэтому здесь мы делаем парадигму и структуру;
сами вожжи (команды/гейты) — это `command-layer` и `scope-reviewer-redesign`. Энфорс — скобками,
внутри скобок работает парадигма.

## Locked decisions (подтверждены Alex)

1. **Язык по слоям.** Harness (`CLAUDE.md`, `memory/`, `agents/`) — английский; проектное знание
   (`kb/`, `decisions.md`) — язык проекта (русский). Английский не улучшает понимание модели;
   критерий — переносимость и поддержка. (См. `decisions.md`, 2026-06-21.)
2. **Два яруса знания, заложить место сейчас.** `kb/` = деривативная вики (с выводами); сырой
   факт-стор (вектор, без выводов) — следующий слой, не строим, но раскладку готовим под него.
3. **CLAUDE.md самодостаточен.** Чтобы понять, как работать, мне хватает одного CLAUDE.md;
   `decisions.md` — архив «почему», в него я лезу только при пересмотре решения. Ничто в правилах
   не должно требовать похода в decisions для понимания.
4. **«Режимов» по сути два: LOOP (дефолт) и Исполнение (по «Выполняй план»).** ambient/`/_n` —
   акценты внутри loop'а; сырьё/вики — ярус KB. Слово «режим» в правила не вносим.

## Структура на диске (целевая, единая для любого проекта)

```
<project>/ai/
  kb/                         деривативная вики: доменные ноты, правятся НА МЕСТЕ, Obsidian (md + [[links]])
                              — будущий сырой вектор-стор монтируется ОТДЕЛЬНО от выводов (см. ниже)
  session/<YYYY-MM-DD-slug>/
      current.md              рабочий док сессии (шаблон ниже)
  decisions/<subject>.md      append-only лог «почему», ПО предмету (не один общий файл)
  plans/<slug>.md             фазированные возобновляемые планы (их проверяет гейт)
```

**Задел под вектор-стор:** сырьё (факты/регуляции/исследования без выводов) НЕ кладётся в `kb/`
вперемешку с вики. Резервируем раздельный путь (напр. `kb/_raw/` как задокументированная точка
монтирования будущего вектор-стора) и фиксируем правило: вики — регенерируемый дериватив, сырьё —
долговечный фундамент, поэтому хранится отдельно и без интерпретаций. Сам стор тут не создаётся.

**Шаблон `current.md`:**

```
# <заголовок сессии> — <дата>
Subject: <slug>
Status: active | parked | closed

## Задача / понимание
(текущая формулировка; переосмысливается по мере loop'а)

## Domain knowledge (собрано в сессии)
(что узнали + указатели в kb/<…>)

## Open questions

## План
(ссылка на plans/<slug>.md, если есть задача-к-решению; иначе — пусто, ambient)

## Решения сессии
(указатели в decisions/<subject>.md)
```

## Re-anchoring spec (Opus — для Sonnet; пофактово, два критерия)

Критерии Alex: **(а) не потерять рабочую специфику** (то, что реально ловит сбои); **(б) каждый
факт должен УКРЕПЛЯТЬ парадигму**, не просто соседствовать. Все 7 фактов переводятся на английский;
суть и специфика сохраняются; в каждый добавляется/уточняется связь с парадигмой:

| Факт | Что сохранить (специфика) | Как укрепляет парадигму |
|------|---------------------------|--------------------------|
| `feedback_directness` | без перформативной похвалы; возражай ДО имплементации | возражение ради лучшего ПОНИМАНИЯ; критика улучшает решение в KB/плане, не льстит чату |
| `feedback_decide_before_building` | обсудить→решить→делать; MORE=LESS; quote-the-yes; остановка=прогресс | это и есть «не исполнять до "Выполняй план"»; loop прежде действия |
| `feedback_opus_plans_only` | Opus планирует, Sonnet строит, ревью вместе | план = доставляемая единица; имплементация — отдельная гейтед-фаза |
| `feedback_scope_agent` | счёт операций; trivial→inline; non-trivial→фазированный план; >3–4 ops=split | план обязан быть фазированным/возобновляемым (вход для проверяющего `scope`) |
| `feedback_communication_expansive` | развёрнутая проза; язык аудитории; без слоганов; ID в «For AI» | KB пишется ДЛЯ читателя, durable, не телеграфно и не в чат |
| `feedback_reviewer_agent` | реальный критик-агент, не самочеклист; один прогон; декомпозиция research | проверка понимания/плана независимым звеном (родственно `scope`-чекеру) |
| `feedback_reusable_tooling` | один закоммиченный документированный скрипт вместо одноразовых | durable тулинг/знание в KB/скрипты, не ephemeral |

Координация: РОЛЬ `scope` («пишет план» → «проверяет план») в `feedback_scope_agent` и
`feedback_reviewer_agent` меняет НЕ этот план, а `scope-reviewer-redesign` (его фаза 3) поверх
английских версий, произведённых здесь. Здесь — только перевод + переякоривание по содержанию как есть.

## Структура боевого CLAUDE.md (что пишем)

Парадигма-спина (английский перевод `CLAUDE.draft.md`, без мета-блока сверху) + краткая строка
«где живёт KB/сессии/решения» (чтобы файл оставался самодостаточным) + `@import`-строки
переякоренных фактов. Гранулярные факты остаются отдельными файлами в `memory/` (granular store,
ведёт `user-profiler`); CLAUDE.md несёт спину и импортирует их. Спина стоит самостоятельно —
импорты дополняют, но не требуются для понимания правил.
